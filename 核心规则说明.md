# DrawLang 核心规则说明

## 1. 设计目标

本课设定义一个简化的绘图语言 `DrawLang`，用于描述基础图形绘制命令。

语言处理流程为：

```text
源程序 -> 词法分析 -> 语法分析 -> 语义分析 -> JSON IR -> 图形绘制
```

本说明文档只定义语言核心规则，包括：

- 完整 EBNF
- 适用于 LR(1) 分析的增广文法
- 产生式编号
- Token 表
- 语义规则
- 错误分类
- JSON IR 约束
- Java 手工 LR(1) 解析器的数据结构约定

不包含工程实现、目录组织、构建命令等内容。

## 2. 基本约定

### 2.1 画布约定

默认画布大小固定为：

```text
WIDTH  = 800
HEIGHT = 600
```

坐标系约定：

- 原点位于左上角
- `x` 轴向右增长
- `y` 轴向下增长

### 2.2 大小写与空白

- 关键字大小写不敏感，`LINE`、`line`、`Line` 均视为同一关键字
- 颜色名大小写不敏感，语义阶段统一转为小写
- 空格、制表符、回车、换行仅作为分隔符，不影响语义
- 一条完整语句必须以分号 `;` 结束


### 2.3 数值规则

- `NUMBER` 支持整数与小数
- 允许正号和负号
- 合法示例：`12`、`0`、`3.14`、`-20`、`+8.5`
- 本版本不支持科学计数法

## 3. 词法规则

### 3.1 关键字

```text
LINE
RECT
CIRCLE
TEXT
COLOR
WIDTH
FILL
NOFILL
SIZE
```

### 3.2 界符与符号

```text
(
)
,
;
```

### 3.3 Token 表

| Token 名称 | 词素示例 | 说明 |
| --- | --- | --- |
| `LINE` | `LINE` | 画线命令关键字 |
| `RECT` | `RECT` | 矩形命令关键字 |
| `CIRCLE` | `CIRCLE` | 圆形命令关键字 |
| `TEXT` | `TEXT` | 文本命令关键字 |
| `COLOR` | `COLOR` | 颜色属性关键字 |
| `WIDTH` | `WIDTH` | 线宽属性关键字 |
| `FILL` | `FILL` | 填充属性关键字 |
| `NOFILL` | `NOFILL` | 无填充关键字 |
| `SIZE` | `SIZE` | 字号属性关键字 |
| `NUMBER` | `12`, `-3.5` | 数值常量 |
| `STRING` | `"hello"` | 字符串常量 |
| `IDENTIFIER` | `red`, `blue` | 标识符，本课设主要用于颜色名 |
| `LPAREN` | `(` | 左括号 |
| `RPAREN` | `)` | 右括号 |
| `COMMA` | `,` | 逗号 |
| `SEMICOLON` | `;` | 语句结束符 |
| `EOF` | 文件结束 | 输入结束 |

### 3.4 Token 识别补充规则

1. 若输入串匹配关键字，则识别为关键字，不识别为 `IDENTIFIER`
2. `STRING` 使用双引号包围
3. 本版本 `STRING` 内不支持转义字符，双引号不能直接出现在字符串内部
4. `IDENTIFIER` 规则为：

```text
[A-Za-z_][A-Za-z0-9_]*
```

5. 当前语言不支持变量，因此普通 `IDENTIFIER` 仅允许出现在颜色位置

## 4. 完整 EBNF

### 4.1 程序结构

```ebnf
program        = { statement } EOF ;

statement      = line_stmt
               | rect_stmt
               | circle_stmt
               | text_stmt ;
```

### 4.2 终结符相关

```ebnf
number         = NUMBER ;
string         = STRING ;
identifier     = IDENTIFIER ;
color_name     = identifier ;
```

### 4.3 LINE 语句

```ebnf
line_stmt      = "LINE" "(" number "," number "," number "," number ")"
                 "COLOR" color_name
                 "WIDTH" number
                 ";" ;
```

### 4.4 RECT 语句

```ebnf
rect_stmt      = "RECT" "(" number "," number "," number "," number ")"
                 "COLOR" color_name
                 "WIDTH" number
                 rect_fill_part
                 ";" ;

rect_fill_part = "FILL" color_name
               | "NOFILL" ;
```

### 4.5 CIRCLE 语句

```ebnf
circle_stmt      = "CIRCLE" "(" number "," number "," number ")"
                   "COLOR" color_name
                   "WIDTH" number
                   circle_fill_part
                   ";" ;

circle_fill_part = "FILL" color_name
                 | "NOFILL" ;
```

### 4.6 TEXT 语句

```ebnf
text_stmt      = "TEXT" "(" number "," number "," string ")"
                 "COLOR" color_name
                 "SIZE" number
                 ";" ;
```

## 5. LR(1) 用增广文法

为了构造手工 LR(1) 分析表，采用不含 EBNF 语法糖的上下文无关文法，并增加起始产生式。

### 5.1 终结符集合

```text
LINE RECT CIRCLE TEXT COLOR WIDTH FILL NOFILL SIZE
NUMBER STRING IDENTIFIER
LPAREN RPAREN COMMA SEMICOLON EOF
```

说明：

- 文法中用 `LPAREN` 表示 `(`
- 用 `RPAREN` 表示 `)`
- 用 `COMMA` 表示 `,`
- 用 `SEMICOLON` 表示 `;`
- 用 `EOF` 表示输入结束

### 5.2 非终结符集合

```text
S' Program StmtList Stmt
LineStmt RectStmt CircleStmt TextStmt
RectFill CircleFill
Number String ColorName
```

### 5.3 增广文法与产生式编号

```text
(0)  S'         -> Program
(1)  Program    -> StmtList EOF
(2)  Program    -> EOF
(3)  StmtList   -> StmtList Stmt
(4)  StmtList   -> Stmt
(5)  Stmt       -> LineStmt
(6)  Stmt       -> RectStmt
(7)  Stmt       -> CircleStmt
(8)  Stmt       -> TextStmt
(9)  LineStmt   -> LINE LPAREN Number COMMA Number COMMA Number COMMA Number RPAREN COLOR ColorName WIDTH Number SEMICOLON
(10) RectStmt   -> RECT LPAREN Number COMMA Number COMMA Number COMMA Number RPAREN COLOR ColorName WIDTH Number RectFill SEMICOLON
(11) RectFill   -> FILL ColorName
(12) RectFill   -> NOFILL
(13) CircleStmt -> CIRCLE LPAREN Number COMMA Number COMMA Number RPAREN COLOR ColorName WIDTH Number CircleFill SEMICOLON
(14) CircleFill -> FILL ColorName
(15) CircleFill -> NOFILL
(16) TextStmt   -> TEXT LPAREN Number COMMA Number COMMA String RPAREN COLOR ColorName SIZE Number SEMICOLON
(17) Number     -> NUMBER
(18) String     -> STRING
(19) ColorName  -> IDENTIFIER
```

### 5.4 关于左递归的说明

`StmtList -> StmtList Stmt | Stmt` 保留左递归是有意为之。

原因：

- 递归下降不适合左递归
- LR(1) 天然适合处理左递归
- 左递归形式更利于按输入顺序逐步归约出语句列表

### 5.5 FIRST 集使用约定

后续手工计算 LR(1) 项目集时，按以下记号处理：

- `FIRST(Number) = { NUMBER }`
- `FIRST(String) = { STRING }`
- `FIRST(ColorName) = { IDENTIFIER }`
- `FIRST(RectFill) = { FILL, NOFILL }`
- `FIRST(CircleFill) = { FILL, NOFILL }`
- `FIRST(Stmt) = { LINE, RECT, CIRCLE, TEXT }`
- `FIRST(StmtList) = { LINE, RECT, CIRCLE, TEXT }`
- `FIRST(Program) = { LINE, RECT, CIRCLE, TEXT, EOF }`

本语言没有空产生式，因此 FIRST 集计算较简单。

## 6. LR(1) 归约语义约定

构造 Java 表驱动分析器时，每次归约不仅要弹栈，还要同步构造语法树节点。

### 6.1 归约产物

| 产生式编号 | 归约结果 |
| --- | --- |
| `(1)` | 以非空 `StmtList` 构造 `ProgramNode` |
| `(2)` | 以空命令列表构造 `ProgramNode` |
| `(3)` | 将右侧 `Stmt` 追加到已有 `StmtList` |
| `(4)` | 新建仅含一个 `Stmt` 的 `StmtList` |
| `(5)~(8)` | 将具体命令节点提升为 `Stmt` |
| `(9)` | 构造 `LineNode` |
| `(10)` | 构造 `RectNode` |
| `(11)` | 返回带颜色的填充描述 |
| `(12)` | 返回无填充描述 |
| `(13)` | 构造 `CircleNode` |
| `(14)` | 返回带颜色的填充描述 |
| `(15)` | 返回无填充描述 |
| `(16)` | 构造 `TextNode` |
| `(17)` | 将 `NUMBER` token 转为数值包装对象 |
| `(18)` | 将 `STRING` token 转为字符串包装对象 |
| `(19)` | 将 `IDENTIFIER` token 转为颜色名包装对象 |

### 6.2 建议的语义值类型

Java 中可以用一个统一的 `AstValue` 体系承载归约值，建议包括：

```text
ProgramNode
List<CommandNode>
CommandNode
LineNode / RectNode / CircleNode / TextNode
Double
String
FillSpec
Token
```

其中：

- `Number -> NUMBER` 归约后建议得到 `Double`
- `String -> STRING` 归约后建议得到去引号后的 `String`
- `ColorName -> IDENTIFIER` 归约后建议得到标准化前的原始颜色名字符串
- `RectFill` 和 `CircleFill` 归约后建议得到 `FillSpec`

## 7. 语句格式示例

### 7.1 合法示例

```text
LINE(50, 50, 300, 100) COLOR red WIDTH 2;

RECT(100, 120, 200, 80) COLOR blue WIDTH 3 FILL yellow;

CIRCLE(400, 250, 60) COLOR black WIDTH 2 NOFILL;

TEXT(200, 500, "DrawLang Demo") COLOR green SIZE 24;
```

### 7.2 非法示例

```text
LINE(50, 50, 300) COLOR red WIDTH 2;
```

参数缺失，属于语法错误。

```text
RECT(10, 20, 100, 50) COLOR cyan WIDTH 2 FILL pink;
```

若 `cyan` 不在允许颜色集合内，则属于语义错误。

```text
TEXT(20, 30, "hello") COLOR red SIZE 0;
```

字号不大于 0，属于语义错误。

## 8. 语义规则

### 8.1 通用规则

1. 所有图形命令必须满足文法定义的参数数量
2. 所有出现在数值位置的内容必须是 `NUMBER`
3. 所有颜色名必须属于允许颜色集合
4. 所有需要正值的尺寸参数必须严格大于 `0`
5. 所有图形必须处于画布有效范围内

### 8.2 合法颜色集合

```text
black
white
red
green
blue
yellow
gray
orange
purple
pink
```

语义分析阶段对颜色名统一转为小写后再检查。

### 8.3 各语句专属规则

#### LINE

- `lineWidth > 0`
- `x1, y1, x2, y2` 必须在画布范围内

#### RECT

- `width > 0`
- `height > 0`
- `lineWidth > 0`
- 左上角坐标 `(x, y)` 必须在画布范围内
- 右下角坐标 `(x + width, y + height)` 不得超出画布
- 若使用 `FILL color`，填充颜色必须合法

#### CIRCLE

- `radius > 0`
- `lineWidth > 0`
- 圆心 `(cx, cy)` 必须在画布范围内
- 圆的边界必须完整落在画布内：
  - `cx - radius >= 0`
  - `cy - radius >= 0`
  - `cx + radius <= WIDTH`
  - `cy + radius <= HEIGHT`
- 若使用 `FILL color`，填充颜色必须合法

#### TEXT

- `fontSize > 0`
- `(x, y)` 必须在画布范围内
- `content` 不能为空串

## 9. 坐标合法性规则

为统一两套前端和后端的行为，定义画布边界为闭区间：

```text
0 <= x <= 800
0 <= y <= 600
```

但对有面积的图形，还需满足完整落入画布，而不是仅起点落入画布。

## 10. 错误分类与输出要求

### 10.1 错误分类

- 词法错误：出现非法字符、非法字符串、无法识别的 token
- 语法错误：token 序列不符合文法规则
- 语义错误：文法正确但含义非法

### 10.2 建议输出格式

两套实现统一采用如下格式：

```text
[Error][Lexical] line 3, col 15: illegal character '@'
[Error][Syntax ] line 5, col 1 : unexpected token SEMICOLON, expected NUMBER
[Error][Semantic] line 7, col 20: radius must be greater than 0
```

要求：

1. 必须包含错误类别
2. 必须包含行号和列号
3. 必须包含尽量明确的错误原因
4. 检测到错误后可立即停止，也可继续恢复分析；基础版本允许报首错即停止

对于 LR(1) 表驱动分析器，语法错误的最小要求是：

- 报出当前状态下无法接受当前输入 token
- 尽可能列出该状态下可接受的 token 集合

## 11. AST 约束

AST 采用扁平命令列表结构：

```text
ProgramNode
└── List<CommandNode>
    ├── LineNode
    ├── RectNode
    ├── CircleNode
    └── TextNode
```

各节点字段如下：

### 11.1 LineNode

```text
x1, y1, x2, y2, color, lineWidth
```

### 11.2 RectNode

```text
x, y, width, height, color, lineWidth, fill
```

其中 `fill` 可为：

- 填充颜色字符串
- `null`

### 11.3 CircleNode

```text
cx, cy, radius, color, lineWidth, fill
```

### 11.4 TextNode

```text
x, y, content, color, fontSize
```

### 11.5 FillSpec

建议单独定义填充描述结构：

```text
FillSpec
├── boolean hasFill
└── String fillColor
```

约定：

- `NOFILL` 归约得到 `hasFill = false, fillColor = null`
- `FILL color` 归约得到 `hasFill = true, fillColor = color`

## 12. Java 手工 LR(1) 解析器数据结构

本节用于冻结 Java 版手工 LR(1) 解析器的核心数据结构，后续实现与报告均以此为准。

### 12.1 符号枚举

建议将语法分析用符号和词法 token 类型分开。

```text
TokenType        // 词法分析输出
GrammarSymbol    // LR 表中的终结符/非终结符
```

其中：

- `TokenType` 用于 Lexer
- `GrammarSymbol` 用于 ACTION/GOTO 查询

### 12.2 产生式结构

```java
class Production {
    int id;
    NonTerminal left;
    int rightSize;
    String description;
}
```

说明：

- `id` 对应本文件中的产生式编号
- `left` 为归约后的非终结符
- `rightSize` 用于归约时弹栈
- `description` 便于调试与报告打印

### 12.3 ACTION 表项

```java
enum ActionType {
    SHIFT,
    REDUCE,
    ACCEPT,
    ERROR
}

class ActionEntry {
    ActionType type;
    int value;
}
```

说明：

- 当 `type = SHIFT` 时，`value` 表示移进目标状态
- 当 `type = REDUCE` 时，`value` 表示归约产生式编号
- 当 `type = ACCEPT` 时，`value` 可固定为 `0`
- 当 `type = ERROR` 时，`value` 可固定为 `-1`

### 12.4 LR 表结构

建议使用两张表：

```java
Map<Integer, Map<GrammarSymbol, ActionEntry>> actionTable;
Map<Integer, Map<NonTerminal, Integer>> gotoTable;
```

若追求运行时效率，也可改为二维数组：

```java
ActionEntry[][] actionTable;
int[][] gotoTable;
```

在课设中更推荐先用 `Map`：

- 可读性更高
- 便于静态写入
- 报错时更方便打印状态可接受集合

### 12.5 分析栈元素

建议将状态栈和语义值栈分离。

```java
Deque<Integer> stateStack;
Deque<Object> valueStack;
```

如需更强可读性，也可定义：

```java
class StackValue {
    GrammarSymbol symbol;
    Object value;
    Token token;
}
```

并使用：

```java
Deque<StackValue> symbolStack;
Deque<Integer> stateStack;
```

推荐方案：

- `stateStack` 单独保存状态
- `valueStack` 仅保存语义值
- 当前输入 token 单独用 `lookahead` 保存

这样实现最直接。

### 12.6 语义值对象

建议统一约定归约值类型：

| 非终结符 | 语义值类型 |
| --- | --- |
| `Program` | `ProgramNode` |
| `StmtList` | `List<CommandNode>` |
| `Stmt` | `CommandNode` |
| `LineStmt` | `LineNode` |
| `RectStmt` | `RectNode` |
| `CircleStmt` | `CircleNode` |
| `TextStmt` | `TextNode` |
| `RectFill` | `FillSpec` |
| `CircleFill` | `FillSpec` |
| `Number` | `Double` |
| `String` | `String` |
| `ColorName` | `String` |

### 12.7 解析器主类建议

```java
class LRParser {
    List<Token> tokens;
    int cursor;
    Token lookahead;

    Deque<Integer> stateStack;
    Deque<Object> valueStack;

    Map<Integer, Map<GrammarSymbol, ActionEntry>> actionTable;
    Map<Integer, Map<NonTerminal, Integer>> gotoTable;
    Map<Integer, Production> productions;

    ProgramNode parse();
}
```

### 12.8 解析主循环

逻辑固定为：

1. 取栈顶状态 `s`
2. 将当前 `lookahead` 映射为终结符
3. 查询 `ACTION[s, lookahead]`
4. 若为 `SHIFT t`
   - 压入目标状态 `t`
   - 将 token 对应的词法值压入 `valueStack`
   - 前进到下一个 token
5. 若为 `REDUCE p`
   - 按产生式右部长度弹出状态
   - 从 `valueStack` 弹出对应个数的语义值
   - 执行归约语义动作，得到新值
   - 查询 `GOTO[topState, leftNonTerminal]`
   - 压入新状态
   - 将新值压入 `valueStack`
6. 若为 `ACCEPT`
   - 返回 `ProgramNode`
7. 若为 `ERROR`
   - 抛出语法异常

### 12.9 语法异常结构

```java
class ParseException extends RuntimeException {
    int line;
    int column;
    TokenType actual;
    List<TokenType> expected;
}
```

这样可以在报错时直接生成：

```text
[Error][Syntax ] line 5, col 1: unexpected token SEMICOLON, expected NUMBER
```

## 13. JSON IR 约束

统一中间表示采用 JSON，对两套前端完全一致。

### 13.1 顶层结构

```json
{
  "commands": []
}
```

### 13.2 各命令的字段约束

#### line

```json
{
  "type": "line",
  "x1": 50,
  "y1": 50,
  "x2": 300,
  "y2": 100,
  "color": "red",
  "lineWidth": 2
}
```

#### rect

```json
{
  "type": "rect",
  "x": 100,
  "y": 120,
  "width": 200,
  "height": 80,
  "color": "blue",
  "lineWidth": 3,
  "fill": "yellow"
}
```

或：

```json
{
  "type": "rect",
  "x": 100,
  "y": 120,
  "width": 200,
  "height": 80,
  "color": "blue",
  "lineWidth": 3,
  "fill": null
}
```

#### circle

```json
{
  "type": "circle",
  "cx": 400,
  "cy": 250,
  "radius": 60,
  "color": "black",
  "lineWidth": 2,
  "fill": null
}
```

#### text

```json
{
  "type": "text",
  "x": 200,
  "y": 500,
  "content": "DrawLang Demo",
  "color": "green",
  "fontSize": 24
}
```

### 13.3 IR 统一要求

1. `type` 一律使用小写
2. 颜色名一律使用小写
3. 无填充统一写成 `"fill": null`
4. 数值统一按 JSON number 输出
5. 两套前端对同一输入生成的字段名、字段值、空值表现必须一致

## 14. 版本边界

当前基础版本明确不支持：

- 变量
- 表达式
- `if`
- `repeat`
- 函数
- 嵌套语句
- 自定义画布大小
- 多文件输入

后续如需扩展，应先修改本规范，再修改两套实现。
