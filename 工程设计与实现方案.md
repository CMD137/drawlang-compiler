# DrawLang 工程设计与实现方案

## 1. 项目定位

本项目用于完成编译原理课程设计，题目为“绘图语言编译器设计与实现”。

根据任务书要求，项目提供两套实现方案：

1. 使用通用程序设计语言手工编写完整解释器
2. 使用 `LEX/YACC` 自动生成词法和语法分析程序

在当前实现计划中：

- 手工方案采用 `Java + 手写 LR(1) 表驱动语法分析`
- 自动方案采用 `Flex + Bison`
- 两套方案共用相同的语义规则和 JSON IR 结构

## 2. 平台与技术路线

### 2.1 运行平台

- 操作系统：Windows

### 2.2 技术选型

- 手工实现方案：Java
- 手工语法分析方法：预计算 LR(1) 分析表，静态写入 Java 程序
- 自动生成方案：Flex/Bison
- 图形绘制后端：Python + matplotlib
- 统一中间表示：JSON IR

## 5. 模块设计

### 5.1 Java 手工前端

目标：

- 手工完成词法分析
- 手工完成 LR(1) 表驱动语法分析
- 构建 AST
- 执行语义检查
- 生成 JSON IR

```text

ParseException
SemanticException

CommandNode
LineNode
RectNode
CircleNode
TextNode
FillSpec
SemanticChecker
JsonEmitter
Main
```

### 5.2 Java 解析器核心数据结构

建议采用如下结构：

```java
enum ActionType {
    SHIFT,
    REDUCE,
    ACCEPT,
    ERROR
}
```

```java
class ActionEntry {
    ActionType type;
    int value;
}
```

```java
class Production {
    int id;
    NonTerminal left;
    int rightSize;
    String description;
}
```

```java
class LRParsingTable {
    Map<Integer, Map<GrammarSymbol, ActionEntry>> actionTable;
    Map<Integer, Map<NonTerminal, Integer>> gotoTable;
    Map<Integer, Production> productions;
}
```

```java
class LRParser {
    List<Token> tokens;
    int cursor;
    Token lookahead;

    Deque<Integer> stateStack;
    Deque<Object> valueStack;

    LRParsingTable table;

    ProgramNode parse();
}
```

### 5.3 Java 解析流程

主流程如下：

1. Lexer 生成 token 序列
2. LRParser 读取 token 序列
3. 根据 `ACTION/GOTO` 表执行移进、归约、接受
4. 在归约动作中构建 AST
5. 语义分析器检查 AST
6. JSON 导出器生成统一 IR

### 5.4 Flex/Bison 自动前端

目标：

- 用 Flex 生成词法分析器
- 用 Bison 生成语法分析器
- 构建 AST
- 执行语义检查
- 生成 JSON IR

建议模块划分：

```text
lexer.l

ast.h / ast.c
semantic.h / semantic.c
json_emitter.h / json_emitter.c
main.c
```

### 5.5 Python 后端

## 6. LR(1) 分析表来源约定

手工 Java 前端虽然在程序中静态写入分析表，但必须保留分析表来源。

建议保留以下材料：

1. 增广文法
2. 产生式编号
3. FIRST 集
4. LR(1) 项目集族
5. ACTION 表
6. GOTO 表

推荐做法：

- 在 `docs/reports/` 中保存整理后的表格
- 如有需要，在 `scripts/` 中写辅助脚本计算项目集和分析表

这样有两个好处：

- 报告中可直接引用
- 若文法修改，可重新核对分析表是否需要更新

## 10. 风险与应对

### 10.1 Windows 工具链风险

风险：

- `win_flex_bison`、C 编译器未配置好，自动前端无法生成和编译

应对：

- 尽早安装并验证 `win_flex`、`win_bison`、`gcc`

### 10.2 LR(1) 表维护风险

风险：

- 文法轻微修改也可能导致分析表整体变化

应对：

- 尽早冻结文法
- 保留分析表来源材料
- 在代码中显式写产生式编号，避免魔法数字失控

### 10.3 两套实现不一致

风险：

- 两边报错口径和 IR 格式可能出现细微差异

应对：

- 一切以 `核心规则说明.md` 为准
- 做双前端 IR 对比测试

### 10.4 中文文本显示风险

风险：

- matplotlib 默认字体可能无法正常显示中文

应对：

- 后端显式设置 Windows 中文字体，如 `SimSun` 或 `Microsoft YaHei`
